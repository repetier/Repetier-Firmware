19 /* ===================== IMPORTANT ========================
20 
21 The LCD and Key support is new. I tested everything as good as possible,
22 but some combinations may not work as supposed.
23 The I2C methods rely on a stable I2C connection. Noise may cause wrong signals
24 which can cause the firmware to freeze.
25 
26 The ui adds quite some code, so AVRs with 64kB ram (Sanguino, Gen6) can not handle all features
27 of the firmware at the same time. You have to disable some features to gain the
28 ram needed. What should work:
29 - No sd card - the sd card code is quite large.
30 - No keys attached - The longest part is the menu handling.
31 - EEPROM_MODE 0 .
32 
33 Currently supported hardware:
34 
35 *** Displays ***
36 
37 - direct connected lcd with 4 data lines
38 - connected via i2c
39 
40 *** Keys ***
41 
42 - rotary encoder
43 - push button
44 - key matrix up to 4x4
45 - rotary encoder via i2c (only slow turns are captured correct)
46 - push button via i2c
47 
48 *** Buzzer ***
49 
50 - directly connected, high = on
51 - connected via i2c, low = on

276 /* #######################################################################
277  Key definitions
278 
279 The firmware is very flexible regarding your input methods. You can use one
280 or more of the predefined key macros, to define a mapper. If no matching mapper
281 is available, you can add you c-code for mapping directly into the keyboard
282 routines. The predefined macros do the same, just hiding the code behind it.
283 
284 For each key, two seperate parts must be defined. The first is the initialization
285 which must be added inside uiInitKeys() and the second ist a testing routine.
286 These come into uiCheckKeys() or uiCheckSlowKeys() depending on the time needed
287 for testing. If you are in doubt, put it in uiCheckSlowKeys().
288 uiInitKeys() is called from an interrupt controlling the extruder, so only
289 fast tests should be put there.
290 The detect methods need an action identifier. A list of supported ids is found
291 at the beginning of ui.h It's best to use the symbol name, in case the value changes.
292 
293 1. Simple push button connected to gnd if closed on a free arduino pin
294  init -> UI_KEYS_INIT_BUTTON_LOW(pinNumber);
295  detect -> UI_KEYS_BUTTON_LOW(pinNumber,action);
296 
297 2. Simple push button connected to 5v if closed on a free arduino pin
298  init -> UI_KEYS_INIT_BUTTON_HIGH(pinNumber);
299  detect -> UI_KEYS_BUTTON_HIGH(pinNumber,action);
300 
301 3. Click encoder, A/B connected to gnd if closed.
302  init -> UI_KEYS_INIT_CLICKENCODER_LOW(pinA,pinB);
303  detect -> UI_KEYS_CLICKENCODER_LOW(pinA,pinB);
304  or UI_KEYS_CLICKENCODER_LOW_REV(pinA,pinB); // reverse direction
305  If you can move the menu cursor without a click, just be adding some force in one direction,
306  toggle the _REV with non _REV and toggle pins.
307  If the direction is wrong, toggle _REV with non _REV version.
308  For the push button of the encoder use 1.
309 
310 4. Click encoder, A/B connected to 5V if closed.
311  init -> UI_KEYS_INIT_CLICKENCODER_HIGH(pinA,pinB);
312  detect -> UI_KEYS_CLICKENCODER_HIGH(pinA,pinB);
313  or UI_KEYS_CLICKENCODER_HIGH_REV(pinA,pinB); // reverse direction
314  If you can move the menu cursor without a click, just be adding some force in one direction,
315  toggle the _REV with non _REV and toggle pins.
316  If the direction is wrong, toggle _REV with non _REV version.
317  For the push button of the encoder use 2.
318 
319 5. Maxtrix keyboard with 1-4 rows and 1-4 columns.
320  init -> UI_KEYS_INIT_MATRIX(r1,r2,r3,r4,c1,c2,c3,c4);
321  detect -> UI_KEYS_MATRIX(r1,r2,r3,r4,c1,c2,c3,c4);
322  In addition you have to set UI_MATRIX_ACTIONS to match your desired actions.
323 
324 ------- Keys connected via I2C -------------
325 
326 All keys and the buzzer if present must be on a connected to a single PCF8574 chip!
327 As all I2C request take time, they belong all in uiCheckSlowKeys.
328 Dont use the pin ids but instead _BV(pinNumber0_7) as pin id. 0 = First pin
329 
330 6. Click encoder, A/B connected to gnd if closed.
331  init -> not needed, but make sure UI_HAS_I2C_KEY is not commented out.
332  detect -> UI_KEYS_I2C_CLICKENCODER_LOW(pinA,pinB);
333  or UI_KEYS_I2C_CLICKENCODER_LOW_REV(pinA,pinB); // reverse direction
334  If you can move the menu cursor without a click, just be adding some force in one direction,
335  toggle the _REV with non _REV and toggle pins.
336  If the direction is wrong, toggle _REV with non _REV version.
337  For the push button of the encoder use 7.
338  NOTICE: The polling frequency is limited, so only slow turns are captured correct!
339 
340 7. Simple push button connected to gnd if closed via I2C on a PCF8574
341  init -> not needed, but make sure UI_HAS_I2C_KEY is not commented out.
342  detect -> UI_KEYS_I2C_BUTTON_LOW(pinNumber,action);
343 
344 -------- Some notes on actions -------------
345 
346 There are three kinds of actions.
347 
348 Type 1: Immediate actions - these are execute and forget actions like home/pre-heat
349 Type 2: Parameter change action - these change the mode for next/previous keys. They are valid
350  until a new change action is initiated or the action is finished with ok button.
351 Type 3: Show menu action. These actions have a _MENU_ in their name. If they are executed, a new
352  menu is pushed on the menu stack and you see the menu. If you assign these actions directly
353  to a key, you might not want this pushing behaviour. In this case add UI_ACTION_TOPMENU to the
354  action, like UI_ACTION_TOPMENU+UI_ACTION_MENU_XPOSFAST. That will show the menu as top-menu
355  closing all othe submenus that were open.
356 
357  ####################################################################### */
